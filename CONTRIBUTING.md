# Contributing to Doorman

Thank you for your interest in contributing to Doorman. This document explains the workflow and standards for changes, issues, and pull requests.

## How to Contribute

1. Fork the repository.
2. Create a branch for your change:
   - `feature/<short-description>`
   - `fix/<short-description>`
   - `docs/<short-description>`
3. Make your changes in a clean and focused commit.
4. Add or update documentation when relevant.
5. Open a pull request against `main`.

## Code Guidelines

- Keep changes small and focused.
- Use clear, descriptive names for files, classes, and methods.
- Follow Kotlin conventions for file and class naming.
- Prefer simple, testable implementations.

## Documentation

- Update `README.md` for any change that affects setup, architecture, or usage.
- Add or update diagrams in `docs/uml/` if the system design changes.
- Document new features or release notes in `docs/release.md` when relevant.

## Testing

- Add tests for new behaviors whenever possible.
- Validate that existing tests still pass.
- Use mocks for hardware-specific flows when necessary.

## Issue Reporting

- Open an issue for bugs, feature suggestions, or documentation improvements.
- Provide a concise description of the problem or request.
- Include steps to reproduce or a short example when applicable.
